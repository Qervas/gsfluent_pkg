1封面

物理仿真出来是粗的，渲染要的是细的，fuse 负责把两边对上。

开场 30 秒

今天讲 gsfluent 流水线里中间这一步 —— fuse。MPM 仿真算出来的是粒子位置，浏览器要的是完整可渲染的 3DGS。fuse 就是把这两边对接起来的桥梁。

关键词

上采样 蒙皮 参考 3DGS

2输入与输出

fuse 吃两套数据：静止状态的视觉真值 + 每帧的物理演化。
类比：参考 ply 是"演员的样貌"，sim ply 是"演员每秒的姿势"。fuse 把样貌套到姿势上，生成完整的"演员表演"。

容易被问

Q：为什么不让仿真直接输出可渲染的 3DGS？
A：仿真不知道颜色 —— MPM 只跟踪位置、速度、形变。颜色跟物理方程无关，硬塞进去只会拖慢仿真。

3流水线位置

fuse 是数据流的"翻译器"，让物理模块和渲染模块各自独立演化。

讲解要点

  • 仿真 → fuse → npz → 浏览器，四个独立阶段
  • 每个阶段只做自己擅长的事
  • 换成别的物理引擎（PBD、SPH），fuse 不用改
类比：像菜谱与厨师 —— 仿真给食材清单，渲染负责摆盘，fuse 是中间的中央厨房。

4为什么需要 fuse

200k 物理粒子 vs 683k 渲染 splat，数量对不上要桥接。

讲解要点

  • 仿真 200k 是因为 O(N²) 邻居搜索；跑 1M 要 25 倍时间
  • 渲染 683k 是因为原模型训练时的精细密度
  • 直接拿 200k 渲染：丢 70% 视觉细节
类比：Pixar 用 10k 引导曲线驱动 1M 渲染细丝，游戏角色用 100 骨骼驱动 50k 顶点 —— 都是粗物理 + 精渲染的套路。

扩展R10 为什么不需要 particle_F

R10 没有上采样问题 —— 它是视频管线，跟我们这套交互管线架构上根本不同。

R10 的工作方式

R10 的渲染分支（--render_img --compile_video）在仿真进程内部直接渲染成 PNG：

  • 不下采样：仿真跑全部 ~1M 粒子（683k gaussian-bound + ~290k filler）
  • 没有 fuse 步骤：cov 和 rotation 从 GPU 直接拿，零中间文件
  • 1:1 对应：gaussian-bound 粒子就是 reference splat；每个粒子的 cov 就是它对应那个 splat 的当前 cov

关键：R10 隐式用了 F

光栅化器拿到的 cov 已经是 Σ_t = F · Σ_rest · Fᵀ 的结果 —— MPM 在内部就用 F 演化 cov。R10 不需要"提取 F"，因为渲染发生在仿真内部，cov 直接可用。从效果上看 R10 就是 F-skinning，只是 MPM 默默做了。

对比表

R10（官方）workbench（我们）
仿真粒子数~1M（全部）200k（下采样）
Splat 与粒子关系1:1~3.4 splat / 粒子
输出PNG → MP4（视频）per-frame .ply（可交互序列）
Cov 来源直接读粒子 covfuse 重建
耗时~25 分钟~4 分钟
能交互浏览吗不能（视频）能（scrub / 转相机）

为什么我们偏离了 R10

工作台需要交互式回放 —— scrub 时间轴、转相机、对比序列。这跟 PNG → MP4 是两个世界：

  1. 要可交互 → 必须持久化每帧 3DGS 数据（.ply / .npz）
  2. 要快 → 用户不能等 25 分钟一次仿真 → 下采样到 200k
  3. 下采样 → 粒子数 ≠ splat 数 → 需要上采样
  4. 上采样 → K-NN ghost → particle_F → 形状粗 → F-skinning（下一步）

每一步都是"为了交互"付的代价。R10 用紧耦合的内部渲染省掉了所有中间问题，但代价是每次重跑才能换角度。这跟"交互"是反的。

扩展F 是什么 —— 形变梯度

F 是 3×3 矩阵，描述材料从静止到当前的所有形变（拉伸 + 旋转 + 剪切）。

数学定义

F = ∂x / ∂X，其中 x 是当前位置，X 是 rest 位置。把"无穷小材料块"从 rest 映射到 current。

在 MPM 求解器里

  • 每个粒子都跟踪自己的 F（MPM 的核心物理量）
  • F 决定本构方程里的应力：σ = f(F)
  • 极分解 F = R · U：R 是纯旋转，U 是对称拉伸
  • 协方差演化：Σ_t = F · Σ_rest · Fᵀ

"particle_F" 这个命名其实有点误导

阶段实际用的
设计意图仿真的形变梯度 F
当前实现仿真的当前 cov（已经是 F 作用过的结果）
F-skinning（下一步）真正提取 F，作用到 splat 自己的 rest cov

所以严格说，当前 particle_F 路径用的是粒子 cov，不是粒子 F。名字是 codename，沿用至今 —— 真正名副其实是 F-skinning 那一步。

讲解技巧

如果听众没学过连续介质力学，可以这样讲："F 就是一个粒子从静止状态到现在被怎么揉过的记录 —— 转了多少、拉了多少、剪了多少，全在这个 3×3 矩阵里。"

5步骤总览

fuse 分 10 步，第 5 步（rest 对应映射）是整个 fuse 最关键的设计决策。

讲解技巧

把第 5 步明确标出来作为"核心"。听众心里有了"主角"，后面看到 K-NN / particle_F 就不会困惑。

6步骤 1：读参考 ply

从参考 ply 把所有视觉属性取下来，仿真不带这些。

容易被问

Q：什么是 SH？
A：球谐函数，存的是跟视角有关的颜色，能渲染镜面反射、半透明这些效果。0 阶就是基础 RGB。
Q：为什么尺度存成 log？
A：3DGS 是神经网络训练出来的，参数要无界实数才好优化。渲染前用 exp 映回正数。

7步骤 2：bbox 归一化

把世界坐标统一映射到 [0, 2]³ 立方体，仿真才能用统一的网格参数。

讲解要点

  • MPM 内部有 N×N×N 背景网格，dx 决定稳定性
  • 不归一化的话，50m 的建筑要 3800 格 → 显存爆炸
  • 归一化后，任何大小的模型都用同一套 grid_lim=2, n_grid=150
类比：像训练神经网络要把图片缩到 256×256 —— 网络结构不用每张图都重新设计。

8步骤 3-4：排序 + 读首帧

读取仿真静止状态，作为整个动画的"零参考点"。

关键术语

rest pose：所有蒙皮算法都需要一个静止姿态作基线，所有形变都相对它计算。

9步骤 5：核心选择 关键

K-NN 平均还是 1-NN 刚性绑定 —— 整个 fuse 最重要的设计决策。

讲解技巧

这页要慢一点讲。听众如果没理解这两条路线的取舍，后面所有内容都会是模糊的。

关键短语（背下来）

  • K-NN 偏过度平滑 → 裂缝被涂抹
  • 1-NN 偏过度刚性 → splat 共享粗形状
  • 对应物理仿真的连续介质 vs 离散单元

10K-NN 路径

每个 ref splat 跟踪 8 个邻居粒子位移的加权平均。
类比：就是角色动画的线性混合蒙皮 (LBS)。一个顶点跟着多个骨骼按权重移动。

容易被问

Q：K=8 真的是最优吗？
A：是经验最佳点。K=4 邻居不够，加权场会断；K=16 邻居跨太远，一个 splat 会"听到"远处的运动 → 鬼影更严重。

11particle_F 1-NN

每个 ref splat 刚性绑到 1 个仿真粒子，带 rest 偏移。
类比：有限元法的标准做法。每个 splat 像一个"挂件"，挂在仿真粒子上跟着移动。

容易被问

Q：为什么要记 rest 偏移？直接吸到粒子位置不行吗？
A：那样 ~3 个共享同粒子的 splat 会塌缩成一个点，细节全丢。带偏移就是让它们刚性地飘在原来的相对位置。

12预生成 rest 模板

用内存换 I/O，每帧只 copy + 覆盖位置/旋转。

数字

  • 参考 ply 161 MB，解析一次约 800ms
  • 151 帧重读 = 121s 纯 I/O
  • numpy memcpy ~20 GB/s，比磁盘 I/O 快 100×
类比：游戏引擎的 vertex buffer —— 顶点上传 GPU 一次，每帧只改 transform，不重新上传。

13每帧融合入口

瓶颈是 eigendecomp（683k × 3×3 矩阵，每帧 150ms）。

容易被问

Q：fuse 一共要多久？
A：约 50 秒（每帧 330ms × 151 帧）。MPM 仿真本身只用 30-40 秒，fuse 反而成了流水线最重的一段。
Q：能不能加速？
A：用 cupy 做批量 SVD 能压到 ~20ms/帧（7× 加速）。暂没做因为整体不是瓶颈。

14K-NN 加权位移

平均"位移"不是"位置" —— LBS 的灵魂。

例子（背下来）

  • 8 邻居静止聚在 (0,0,0)，splat 在 (0.1, 0, 0)
  • 平均位置 → splat 被拉到 (0,0,0)，细节丢
  • 平均位移 → splat 仍在 (0.1, 0, 0)，细节保

数学上：位移对常数项不敏感，叠加不破坏 splat 相对粒子的位置。

15particle_F 刚性绑定（代码）

位置 = 粒子当前 xyz + rest 偏移；旋转/尺度从粒子当前 cov 提取。

关键三步

  1. 位置覆盖（带 rest 偏移）
  2. cov 乘 extent²（坐标系转换）
  3. eigh 分解出 quaternion 和 log_scale

16cov 乘 extent²（数学细节）

cov 是二阶量，比位置多乘一次 extent。

原理

线性变换 x' = A·x 作用在协方差上是 Σ' = A·Σ·Aᵀ。各向同性放缩 A = extent·I，所以位置一阶，cov 二阶。

关键术语

eigendecomposition（特征分解）= PCA 的核心运算。给协方差矩阵，分解出旋转和尺度。3DGS 用 (rot, scale)，分解就是逆向把 cov 拆回这两个。

17坐标变换到输出系

仿真在归一化空间，输出回世界系；可选把 Y-up 转 Z-up。

容易被问

Q：Y-up 和 Z-up 是什么？
A：哪个轴朝上。OpenGL / Blender 默认 Y-up，3D 扫描 / 建筑 / 地理信息默认 Z-up。我们的工作台统一用 Z-up（建筑站立感更自然）。

18原子写入

tmp + rename = POSIX 原子操作，避免 watcher 读半截文件。

常见 bug 故事

直接写 frame_0042.ply，写到一半 watcher 读 → 解析半个 ply → 崩溃。用 os.replace() 一行解决：读者要么看旧文件，要么看完整新文件，不存在中间态。

19npz 缓存

把 151 个 ply 打包成 1 个 npz，浏览器 mmap 切帧零解析。

数字

  • 151 个 ply：23 GB，每次切帧解析 200ms+
  • 1 个 npz：2.8 GB，mmap 切片，零解析

NeRF Studio、Inria 3DGS viewer 都用同样模式。

21核心取舍 K-NN vs particle_F

两种路线都不完美 —— K-NN 在裂缝处出鬼影，particle_F 让 splat 形状变粗。

讲解要点

  • 这是平滑 vs 不连续的本质矛盾
  • K-NN 等价于低通滤波 → 噪声和裂缝都被磨平
  • particle_F 不做平均 → 裂缝正确但失去过度采样优势
类比：用模糊滤镜去噪 → 文字也模糊了。锐化保边 → 噪声放大了。两种取向都不完美。

22F-skinning 下一步

从粒子的形变中提取张量 F，作用到 splat 自己的 rest cov 上 —— 兼顾细节和形变。

核心思路

不直接继承粒子的 cov，而是继承形变。每个 splat 保留自己的 rest 形状，跟着粒子的形变张量走。

F = sqrt(Σ_particle_t · Σ_particle_rest⁻¹)，然后 Σ_splat_t = F · Σ_splat_rest · Fᵀ

23总结

fuse = 坐标桥 + 上采样规则 + 每帧属性覆盖。

视觉效果好坏 ≈ 上采样规则的选择。其余步骤都是工程支撑。

FAQ — 听众可能问的问题

1. 为什么不让 MPM 直接输出可渲染的 3DGS？

三个原因：(1) MPM 不知道颜色，只跟踪位置、速度、形变；硬塞颜色字段会拖慢仿真。(2) MPM 为速度跑 200k，渲染要 683k 的细节密度，本身就不匹配。(3) 模块化：fuse 是纯几何映射，换成 PBD、SPH 也能用同一套。

2. 200k 这个数是怎么定的？

经验值。MPM 的邻居搜索是 O(N²)，跑 1M 粒子约 25 分钟一帧，用户体验崩。200k 是"足够细节 + 用户能等"的拐点。可以通过 --target_particles 调，但不建议低于 100k（视觉损失明显）。

3. 为什么用 KD-tree 不用别的最近邻算法？

scipy 的 cKDTree 是 cython 实现的，构建 + 查询都很快。3D 空间下 KD-tree 是 O(N log N) 构建、O(log N) 单点查询。换成 ANN（近似最近邻）能更快但不必要 —— 我们的 200k × 683k 查询用 KD-tree 1-2 秒搞定。

4. K=8 真的是最优吗？

是经验最佳点。K=4 邻居覆盖不够，加权场会断；K=16 邻居跨远距离，远处的运动也会影响近处的 splat，鬼影更严重。8 在 3D 空间下是 LBS 的常用值（cube 的 8 个顶点直觉）。

5. 为什么反距离权重不用高斯权重？

高斯权重要选带宽 σ，σ 难调，对粒子密度敏感。反距离 1/d 自动按局部密度伸缩，无超参。本质上是 Shepard 插值，GIS 领域常用。

6. particle_F 的 1-NN 会不会也有边界问题？

会，但表现不同。1-NN 在裂缝处不出鬼影，因为没有平均。但它会让 ~3 个 splat 共享同一粒子的形状 → 细节变粗。两种 bug 在视觉上感受很不一样：K-NN 的鬼影是"消失"，particle_F 的形状粗糙是"模糊"。

7. eigendecomp 这么慢，能不能用 SVD？

对对称矩阵，eigh 比 svd 快约 2× —— 因为它利用了对称性。SVD 是通用算法，对任意矩阵都行。我们的 cov 永远对称正定，所以用 eigh。要更快，可以用 cupy 在 GPU 上批量做 —— 暂没做因为整体瓶颈不在这。

8. 为什么 cov 要乘 extent²，但位置只乘 extent？

因为协方差是二阶量。线性变换 A 作用在位置上是一阶 (x' = A·x)，作用在协方差上是二阶 (Σ' = A·Σ·Aᵀ)。对各向同性放缩 A = extent·I，就分别得到 extent 和 extent²。这是仿射变换的标准结果，统计学课都讲。

9. 为什么不直接缓存所有帧到内存？

太大。一个 .npz 2.8 GB；如果同时打开多个 sequence，浏览器内存会爆。npz 用 mmap 让操作系统按需分页 —— 看哪一帧就只加载哪一帧，比手动缓存安全得多。

10. fuse 这一步能不能省掉？

不能，除非接受丢掉视觉细节或者每次仿真等 25 分钟。fuse 解决的是物理采样数和渲染采样数不匹配的问题，这个矛盾是固有的 —— 仿真为速度而粗采样，渲染为质量而精采样。

11. 别人也这么做吗？

是的。Pixar 的毛发系统、Blender 的物理蒙皮、UE5 的 Chaos Physics、Houdini 的 Vellum —— 全都是"粗物理 + 精渲染 + 上采样桥梁"的套路。差别在桥梁的具体算法（LBS、DQS、F-skinning、Wrap 等）。

12. F-skinning 什么时候做？

它是 particle_F 的下一步迭代。当前 particle_F 让 splat 继承粒子的 cov，结果 splat 形状变粗。F-skinning 改成继承粒子的形变，让 splat 保留自己的 rest 形状 + 跟着粒子的形变张量走。预计 1-2 小时实现。

13. 整体流水线一次完整运行要多久？

cluster_6_15（683k splat、200k 仿真粒子、151 帧、earthquake recipe）大约：MPM 仿真 ~140s + fuse 70s + npz 构建 ~20s = 总共约 4 分钟。GPU 用一张 A100。

14. 失败的 sim 怎么排查？

看 /api/runs/<name>/log。常见两类：(1) tensor 形状不匹配 —— 仿真脚本里的 mask 同步没做对。(2) CUDA error 700 —— 数值不稳定，多半是配方参数太激进（cuboid 速度太大、边界跟几何重叠）。

15. F 到底是什么？粒子 cov 又是什么？两个不一样吗？

F 是形变梯度（3×3 矩阵），记录材料点从静止到当前怎么被揉过：旋转 + 拉伸 + 剪切。MPM 求解器内部一直跟踪每个粒子的 F。

cov（协方差）是 F 作用在 rest cov 上的结果：Σ_t = F · Σ_rest · Fᵀ。所以 cov 是 F 的"作用效果"，F 是"原因"。

我们当前 particle_F 路径用的是 cov（直接拿粒子的 cov 给 splat 用）。F-skinning 那一步才会真正从 cov 反推出 F，再作用到 splat 自己的 rest cov 上。

16. R10 那条路怎么不用做 fuse 也不用做 K-NN？

因为 R10 是视频管线，不是交互管线。它在仿真进程内部直接渲染成 PNG，跑全部 ~1M 粒子（不下采样）。每个 gaussian-bound 粒子就是一个 ref splat（1:1 对应），cov 从 GPU 直接拿，渲染完就扔。

我们工作台为了交互回放，把渲染和仿真解耦了：每帧得写 .ply 文件、得在浏览器里渲染、得让用户能切换不同序列 —— 这些都要求数据形态标准化、要求下采样以保速、要求上采样补细节。一连串决定让我们必须做 fuse + K-NN/particle_F。

简而言之：R10 用紧耦合省掉了所有问题，代价是每次重跑才能换角度。我们解耦换来了交互性，代价是引出了一整套上采样问题。

17. 现在的 npz schema (v2) 有哪些字段？以后会变吗？

v2 包含：每帧位置、每帧四元数、静态尺度、静态颜色、静态不透明度。如果要 F-skinning 完整生效，需要 v3 增加"每帧尺度"字段（现在的尺度是静态的）。v3 会让 .npz 增大约 50%。
